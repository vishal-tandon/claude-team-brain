## 2026-06-15 — Session 1 — From "does it have the same bug" to a validated self-installing brain

Vishal resumed from a prior handoff and brought a pointed question: in his work edition of the brain, new skills pushed to the repo were not auto-installing for users, and a fix had been added so every new session picked them up. He asked whether this public-template architecture had the same gap. Claude read the actual skill and manifest code rather than answering from the handoff, and confirmed the gap was real: the docs' "version omission means always-latest" rationale was false, third-party marketplaces default auto-update off, and `setup-brain` never wrote the flag that would make pushed skills arrive. Claude fixed it by having setup write `autoUpdate` into the user's settings, then Vishal asked the larger question that set the session's direction: run a full two-user end-to-end simulation and find where the architecture falls apart.

That simulation surfaced twelve findings, three of them hard blockers (a context-import detector that grepped a token setup never writes, a pre-commit guard that blocked legitimate skill pushes, and private-repo joiners with no access path). Claude proposed a patch plan, and Vishal reframed the north star: a non-technical user should clone, say "set up my brain", and have Claude do everything, no JSON editing, no "learning the brain". That reframe became the spine of the work, `setup-brain` was rewritten from a wizard that reads hand-edited placeholders into an installer that derives identity from the git remote, interviews for two choices, writes its own config, and self-heals on re-run. Two install paths were built (lazy paste-a-link and manual fork-and-open), plus a bootstrap nudge so a cold Claude reliably runs setup.

The turning point was live testing. Vishal insisted on running the install himself in a cold sandbox rather than trusting Claude's doc-verified claims, and he was right to. The first live run exposed that the plugin `source: "."` form, which docs and a research agent had twice confirmed valid, was rejected by the real CLI, and that the manifest push had to happen before the marketplace add because the add fetches from the remote. Subsequent rounds caught more: accept-edits does not cover Bash commands (fixed with a shipped permission allowlist), the SessionStart hook cannot run a skill (only a shell pull), and the optional hook should be default-on. Vishal gave UX feedback each round, less stop-and-go, plainer language, clearer teammate onboarding, and Claude folded each into the skill.

By the end the full lifecycle was proven live: a stranger installs by saying "set up my brain", a skill pushed afterward auto-arrives on the next session (the original bug, now demonstrably dead), disconnect cleanly removes everything including the bare-`@` import line the old code would have missed, and re-running setup reconnects idempotently. Nineteen findings were logged and resolved or noted in `meta/e2e-simulation-findings.md`, a deterministic test suite and an isolated sandbox harness were built, and the work merged to main as PR #1. Left open as secondary: joiner connect with a second identity, the Path 1 fork branch, governed mode, and share/sync drift.
